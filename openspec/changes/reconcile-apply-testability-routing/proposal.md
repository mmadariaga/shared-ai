## Why

`sai/instructions/apply.md:58` defines a Step as non-testable exactly when its body contains no `##### RED phase` block, which makes two other rules in the same file unreachable: the single dispatch's RED → GREEN handling (`apply.md:72-77`) and the report contract's claim that a single-dispatch Step carries real values in both field 3 and field 4 (`apply.md:153`). Meanwhile `sai/instructions/implement.md:274` explicitly promises support for a plan that has RED blocks and no `interfaces.md` — a Step shape the routing rule currently sends to the blind test-writer, whose prompt is built from an `interfaces.md` `## Step N` section that does not exist.

## What Changes

- Restate the testability routing signal in `apply.md` as a **two-part test**: a Step routes to the two-dispatch (blind test-writer → implementation) flow only when its body contains a RED block **and** a `## Step N` contract for that Step is available in `interfaces.md`. Every other Step routes to a single dispatch.
- Make the single dispatch's RED → GREEN handling reachable: a Step with a RED block but no available contract is executed by one subagent that authors the test from the Step's own scenario descriptions and then runs RED → GREEN, exactly as `implement.md:274` promises.
- Re-scope the `interfaces.md` ↔ `implementation.md` Step-N key-integrity guard so it no longer treats a *missing* `## Step N` as an automatic desync STOP. Absence becomes a routing input; an **ambiguous** match (several `## Step N` sections for the same `N`) remains a STOP.
- Require the coordinator to print a visible one-line trace when a Step with a RED block routes to a single dispatch because no contract was available, so a genuine desync stays human-detectable now that absence no longer halts the run.
- Re-key the per-dispatch value table in `## Subagent Report Contract` from Step testability to **dispatch routing**: a single dispatch carries a real value in field 3 exactly when the Step it executes has a RED block, and always carries a real value in field 4.
- Non-breaking for existing plans: a Step with a RED block and a matching contract keeps routing to two dispatches; a Step with no RED block keeps routing to one.

## Additional Notes

- **The contradiction is resolved in favour of the rules, not the signal.** Two independent sections (`apply.md:72-77` and `apply.md:153`) were written on the assumption that a single dispatch can run RED, and `implement.md:274` describes the plan shape that produces it. One under-specified sentence is the more likely defect; it recorded the common case as if it were the whole rule when the test-writer split was introduced.
- **The contract-availability check is per Step, and the per-Step branch is reachable — through desync, not through omission.** `design.md:192` omits a `## Step N` section only for a step that introduces neither a public interface nor a testable assertion; a Step carrying a `##### RED phase` block has a testable assertion by construction and is therefore never omitted, so the generators' omission rule cannot produce this state. The live path is the one the guard's own rationale names: `interfaces.md` is regenerated wholesale while `implementation.md` is preserved byte-for-byte, so an orphan Step carrying a RED block can outlive its contract. Whole-file absence — the case `implement.md:274` names — is the other path, and it is legitimate rather than a fault. The per-Step form covers both; a file-existence test would catch neither. Changing `design.md`'s omission rule remains an explicit non-goal.
- **The two absence shapes differ, and the change treats both non-fatally for different reasons.** Whole-file absence is legitimate and must run. Per-Step absence on a RED-carrying Step is a probable desync — there is no legitimate path that produces it — but the single dispatch executes the Step correctly from its own scenario descriptions either way, so halting the run would cost more than it reports. The fall-back trace line is what makes the second shape visible without paying that cost. The existing guard's STOP is preserved for the one case where execution genuinely cannot proceed correctly — several `## Step N` sections matching the same `N`, where the coordinator would have to guess which contract to inject.
- **Rejected alternatives.** (a) Delete the RED → GREEN handling from the single dispatch and set field 3 to `n/a` in the table — simpler, but it removes the only path for a RED block with no contract and silently breaks `implement.md:274`. (b) Make `interfaces.md` mandatory for every step with a RED block — a larger change with a different blast radius; it constrains `/sai-2-design`'s omission rule and contradicts `implement.md:274` rather than reconciling with it. (c) Leave it to coordinator judgement at dispatch time — the routing decision determines whether a blind subagent is created at all, so it is exactly what must not be improvised.
- **Accepted trade-off.** The routing condition becomes a two-part test and no longer states in one sentence.
- The single dispatch's existing *No exploration* rule already permits reading test files and test infrastructure, so a single dispatch that authors its own test from scenario descriptions needs no new read permission. The blind test-writer's blindness rule, the *No exploration* rule, and every dispatch's prompt contents are non-goals for this change.
