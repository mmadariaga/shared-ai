## Why

When a step in `implementation.md` has no observable browser behavior (a service-side, config, or otherwise non-UI step), `/sai-3-implement` currently improvises a `- [ ] No human check required` checkbox in the Human Verification section. That invented checkbox is neither a real action (`/sai-4-apply` cannot decide whether to mark it) nor ignorable (`sai-archive` STOPs on any `- [ ]` per `archive.md:26`, and `sai-pr` extracts it as a test-plan item per `pr.md:63`). The plan template already solves the sibling "component not yet rendered" case with an italic note (no checkbox) — this change extends that same pattern to the "no UI surface" case and makes `/sai-4-apply` resilient to Human sections that carry zero checkboxes.

Relatedly, `sai-archive`'s Completion Check (`archive.md:25-27`) currently makes a silent, absolute hard STOP on any `- [ ]` in `implementation.md` — the user discovers the block only when they try to archive. This change turns that hard STOP into a soft confirmation gate: `sai-archive` lists the concrete unchecked items and asks before proceeding, aligning it with the pipeline's other soft gates (the `sai-3-implement` FALLO MENOR re-run warning, the `sai-4-apply` UX, the `sai-pr` test plan) and with the Discards confirmation pattern at `implement.md:85`.

## What Changes

- `sai/instructions/implement.md` — document a new **Service-side / non-UI step** archetype in `<plan_template>` and its surrounding Human-check rules: when a step produces no observable human check, the plan SHALL omit the `**Human (...)**` checklist header and emit an italic note instead of any checkbox (mirroring the existing "not yet rendered" note). The invented `- [ ] No human check required` checkbox is prohibited.
- `sai/instructions/apply.md` — the Human Verification gate keys on **checkbox count**: it presents and waits only when the step's Human section contains at least one `- [ ]` checkbox; when the Human section has zero checkboxes (italic note only, or no Human section at all), the gate is skipped and the agent proceeds directly to the commit proposal.
- `sai/instructions/archive.md` — the Completion Check (lines 25–27) changes from a silent hard STOP on `- [ ]` to a **soft confirmation gate**: list every unchecked item (`implementation.md:{line}` + enclosing `#### Step N` + checkbox text), prompt `Continue archiving with N unchecked items? (y/n)`, and archive only on an explicit `y`; on `n` or silence, stop and report. Conversational in chat — no `.openspec.yaml` key, no new formal gate. Only the unchecked-items rule changes; the CORE/AUDIT classification, missing-main-spec handling, and spec-sync checks are untouched.
- `AGENTS.md` — one short blurb under **Critical conventions** documenting the asymmetry: *checkboxes are actions, italic notes are explanations*.
- No wrapper change: `commands/claude/` and `commands/opencode/` are untouched (this edits instruction bodies and specs, not the thin wrappers — the archive logic lives in `sai/instructions/archive.md`, which both wrappers fetch; if any wrapper edit had been needed it would be mirrored in `commands/claude/sai-archive.md` and `commands/opencode/sai-archive.md`).

## Capabilities

### New Capabilities
- `implementation-md-noop-marker`: normative encoding for "no observable human check" in `implementation.md` — omit the Human checklist header and use an italic explanatory note (no checkbox). Introduces the Service-side / non-UI step archetype and forbids the invented `- [ ] No human check required` checkbox.

### Modified Capabilities

A Human section with zero checkboxes (italic note only) skips the gate exactly like a step with no Human section. (Folded into the existing capability rather than a separate `apply-md-skip-empty-gate` spec so the gate's present-and-skip logic stays in one file — splitting it risks the two halves diverging.)

(y/n)`, archiving only on `y`. (This is the behavior the user labelled `archive-md-soft-unchecked-gate`; it is delivered as deltas on this existing capability because the hard STOP it inverts is a normative requirement *in this spec* — a standalone new capability would leave a direct contradiction.)

The "check `implementation.md` not `tasks.md`" and "skip when absent" behaviors are unchanged.

## Additional Notes

- **Two-place fix, single root cause.** The invented checkbox is a symptom of the plan template never naming the "no UI surface" case. Fixing generation (`implement.md`) prevents new bad checkboxes; re-anchoring the apply gate on checkbox count (`apply.md`) makes `/sai-4-apply` robust even if a stray zero-checkbox Human header survives from an older plan or partial adoption.
- **Encoding shape** mirrors the existing not-yet-rendered note exactly: no `**Human (verify in browser before committing):**` header, just an italic parenthetical, e.g. `*(No Human checks — service-side step with no observable browser behavior.)*`. This keeps the Verification Checklist shape uniform across all three no-human-check situations (not-yet-rendered, deferred-away, and service-side).
- **Why the gate keys on checkbox count, not header text.** Keying on the literal `**Human ...**` header would still fire the gate on a header followed by zero checkboxes. Counting `- [ ]` checkboxes in the Human section is the robust invariant: zero → skip, ≥1 → present and wait. This also correctly leaves the `*Deferred from Step N*` blocks gated (they always carry ≥1 checkbox).
- **The `- [ ] No code changes from this audit` checkbox is deliberately excluded.** It is an acknowledgment *action* the user takes to close an all-Discarded audit step; it lives in the audit step body, not a Human Verification section, and is required by `audit-artifact-ingestion`. The noop-marker rule targets only the Human Verification section, so it never applies to this checkbox.
- **Archive soft gate — concrete list is mandatory.** The prompt must name each unchecked item as `implementation.md:{line}` + its `#### Step N` + the checkbox text, not just a count. A bare "N unchecked items" would reproduce the current "silent" complaint — the user needs to see *what* is unmarked to make an informed call.
- **Archive soft gate — chat-only, no formal gate.** Confirmation is conversational: no `.openspec.yaml` approval key, no new formal approval gate — identical to the Discards confirmation pattern (`implement.md:85`). Archive proceeds only on an explicit `y`; `n` or silence stops and reports.
- **Archive soft gate — scoped to the unchecked-items rule.** Only `archive.md:25-27` changes. The CORE/AUDIT classification, AUDIT soft-warning, missing-main-spec handling, and spec-sync checks are untouched, and the change stays inside `sai/instructions/archive.md` (both wrappers fetch it), so the existing `sai-archive-soft-warn-audits` "no wrapper/schema/upstream modification" constraint still holds.
- **Two capability specs carry the archive behavior** because the hard STOP was normatively asserted in two places: `sai-archive-soft-warn-audits` ("Incomplete-tasks hard stop is preserved") owns the primary REMOVE+ADD, and `implementation-progress-tracking` ("archive-tracks-implementation") gets its incomplete-tasks scenario updated so the two specs do not contradict each other after the change.
- **No GLOSSARY.md needed** — "Service-side / non-UI step", "noop marker", "italic note", and "soft confirmation gate" are spec-defined terms local to this change; no new project-wide domain vocabulary is introduced.
