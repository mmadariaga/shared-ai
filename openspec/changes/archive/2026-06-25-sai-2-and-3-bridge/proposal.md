> **⚠ POST-HOC RECORD** — This proposal was backfilled after implementation. It describes a decision already made, not one being proposed.

## Why

The sai-2 design step always ended with a mandatory STOP that forced the user to open a fresh chat before running `/sai-3-implement`. For implementations where the design context is valuable and the same model is wanted, that forced break discarded useful context and offered no way to continue inline. This change lets the user run sai-3 immediately after sai-2, reusing the current model and the in-memory design context, while still keeping the fresh-chat path as the default pipeline option.

## What Changes

- `sai/commands/sai-2-design.md`: the Completion block no longer hard-stops. It now asks the user how to proceed: (a) stop for a new chat with a cheaper `/sai-3-implement` model — the standard pipeline path; or (b) continue now in the same chat, reusing the current model and design context. Choosing (a) preserves the original verbatim STOP+handoff. Choosing (b) gates on user review of `design.md` and `tasks.md`, re-reads both from disk, then follows the shared implement invocation.
- `sai/commands/sai-3-implement.md`: the inline Load instructions / Run / Completion blocks are removed and replaced by a single fetch of the shared `implement-invocation.md`.
- `sai/instructions/implement-invocation.md` (new): a shared neutral invocation point holding the load/run/completion contract for the implementation-plan generator, fetched by both `sai-3-implement` (fresh chat) and the "continue now" branch of `sai-2-design` (same chat).

## Capabilities

### New Capabilities

- `design-implement-bridge`: a same-chat continuation path that lets sai-2 invoke the implementation-plan generator directly, gated by user review, reusing the current model and design context.
- `shared-implement-invocation`: a single shared instruction file that centralizes the implementation-plan generator's load/run/completion contract, fetched by both the fresh-chat (sai-3) and same-chat (sai-2 branch) entry points.

### Modified Capabilities

- `uniform-handoff`: the sai-2 completion no longer unconditionally STOPs with the fresh-chat handoff. It branches; the verbatim STOP+handoff is preserved only on the new-chat path. The sai-3 completion contract is relocated into the shared `implement-invocation.md`.

## Impact

- **New files**: `sai/instructions/implement-invocation.md`
- **Modified files**: `sai/commands/sai-2-design.md`, `sai/commands/sai-3-implement.md`
- **Out of scope**: `design.md`, `tasks.md`, `implementation.md` — not generated by `/sai-backfill`
