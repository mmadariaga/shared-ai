> **⚠ POST-HOC RECORD** — This proposal was backfilled after implementation. It describes a decision already made, not one being proposed.

## Why

Design generation could proceed to `tasks.md` even when `design.md` contained unresolved Open Questions, causing tasks to be scaffolded on incomplete or ambiguous design decisions. The fix blocks forward progress until every question is answered.

The opencode config message in `bin/install.js` was also clarified: instead of instructing users to manually add a section, it now prompts them to verify that the settings are properly configured — more accurate for cases where the config file already contains those keys.

## What Changes

- `sai/instructions/design.md`: adds an Open Questions gate section that requires the agent to present any unresolved questions to the user and collect answers before writing `tasks.md`; `design.md` is updated with those answers before continuing
- `bin/install.js`: cosmetic text clarifications to the opencode config fallback message (intro line and inline comment wording)

## Capabilities

### New Capabilities

- `open-questions-gate`: after writing `design.md`, the agent reviews the Open Questions section and blocks progress to `tasks.md` until every question is resolved and incorporated into `design.md`

### Modified Capabilities

- `opencode-config-message`: the fallback message shown when `opencode.jsonc` already exists now says "Verify that you have these settings properly configured" and uses "Your trusted low-cost model below" for inline comments

## Impact

- **New files**: none
- **Modified files**: `sai/instructions/design.md`, `bin/install.js`
- **Out of scope**: `design.md`, `tasks.md`, `implementation.md` — not generated by `/sai-backfill`
