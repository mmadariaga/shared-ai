## Why

The `<plan_template>` Prerequisites block in `sai/instructions/implement.md` unconditionally creates a new branch "from `main`". This is wrong for stacked work (a feature should branch off its parent, not `main`) and breaks on repositories whose default branch is `master`.

## What Changes

- Add a **branch-base prompt**: when the user selects a *new* branch (option 1 suggested name, or option 3 manual name) that does not yet exist, present a 2-option closed choice — base the new branch on the repository default branch (default) or on the current branch — via the harness option-picker per `remember.md` closed-choice rules.
- **Detect the default branch dynamically** (`main` vs `master`) instead of assuming `main`.
- Replace the hardcoded "create it from `main`" instruction with "create it from the chosen base branch".
- **Skip the base prompt** when: option 2 (stay on current branch) is chosen, the target branch already exists, or the current branch already equals the default branch (the two bases would be identical).
- Preserve the existing three-option branch-selection order and the input-language localization behavior.

### Modified Capabilities
- `sai-apply-branch-prompt`: the branch-creation base is no longer hardcoded to `main`; the plan template gains a branch-base prompt and dynamic default-branch detection, with defined skip conditions.
- `sai-implement-branch-prompt`: its "Branch selection prompt" requirement also hardcodes create-from-`main` (option 3 and the "selected branch does not exist" scenario) — updated to reference the chosen base branch so the spec set does not contradict `sai-apply-branch-prompt`.

## Impact

- **Edited files**: `sai/instructions/implement.md` — only the `## Prerequisites` block inside `<plan_template>`. No other SAI instruction file and no production application code.
- **Spec-only reconciliation**: the `sai-implement-branch-prompt` capability spec is updated in lockstep (delta only) so it no longer hardcodes create-from-`main`. No separate instruction file corresponds to it — both capabilities govern the same `implement.md` plan template.
- **Downstream**: `implementation.md` artifacts generated by `/sai-3-implement` (and applied by `/sai-4-apply`) will record the chosen base branch in their Prerequisites section. Existing `implementation.md` files are untouched.
- No change to `apply.md` git-authorization gate — the base decision is captured as plan text; the git operation itself still runs only under the existing authorization gate.

## Additional Notes

- Current template text (implement.md): three fixed options, then "If the selected branch does not exist, create it from `main` before implementing."
- The `sai-apply-branch-prompt` capability already constrains edits to the `<plan_template>` Prerequisites block; the base prompt lives inside that block, so that scope constraint still holds and is not modified.
- Default base = default branch (main/master); "current branch" is the alternative. Flip the default if the reviewer prefers current-branch basing.
- Suggested detection: resolve `refs/remotes/origin/HEAD` (e.g. `git symbolic-ref --quiet refs/remotes/origin/HEAD`), falling back to whichever of `main`/`master` exists locally. Exact command is an implementation detail for design/implement. Design MUST also define the edge behavior when both `main` and `master` exist, and when neither exists / there is no `origin` (bare or local-only repo).
- The base prompt is a follow-up to the existing three-option prompt, not a replacement — the three options and their order are unchanged.
