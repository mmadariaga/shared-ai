## Why

The `<plan_template>` in `sai/instructions/implement.md` hardcodes a "no main/master + select branch" rule with no opt-out, forcing every generated `implementation.md` to mandate a branch switch. This is rigid for workflows that commit directly to main, work in detached HEAD, or stay on a long-lived feature branch already in progress. The user wants **complete opt-out** — no branch is prohibited, no warning, user's responsibility.

## What Changes

- Replace the `## Prerequisites` block inside `<plan_template>` (lines ~203-208 of `sai/instructions/implement.md`) with a 3-option branch-selection prompt that detects the current git branch and lets the user stay on it.
- The agent MUST detect the current branch via `git rev-parse --abbrev-ref HEAD` (or equivalent) before presenting options.
- Options are presented in this order: (1) suggest `{feature-name}` from the change name, (2) stay on `{current-branch}`, (3) free text for custom name.
- Option labels in the language the user writes in when invoking `/sai-3-implement` (English as fallback when unclear); all surrounding plan text in English.
- If the selected branch does not exist, create it from `main` before implementing.
- No changes to `apply.md` or any other instruction file.
- Existing `implementation.md` files are not regenerated.

## Capabilities

### New Capabilities
- `sai-apply-branch-prompt`: Branch-selection step baked into every future `implementation.md` plan generated from `<plan_template>`. Detects the current git branch at plan-generation time and presents a 3-option prompt (suggest, stay, free text) with **complete opt-out** — the user can stay on any branch including main, master, or detached HEAD. No prohibition, no warning.

### Modified Capabilities
(none)

## Impact

- **Instruction file**: `sai/instructions/implement.md` — `## Prerequisites` section inside `<plan_template>` only.
- **Downstream artifacts**: All future `implementation.md` files generated by `/sai-3-implement` will use the new 3-option prompt instead of the hardcoded 2-option rule.
- **No code, config, or CI changes.**
- **No change to `/sai-4-apply`** — apply inherits branch info from the plan.

## Additional Notes

- **Relationship with sibling spec**: `openspec/specs/sai-implement-branch-prompt/spec.md` already defines a similar branch prompt at `/sai-3-implement` runtime. This new spec is an **alternative**, not complementary — it lives in the `<plan_template>` and runs at `/sai-4-apply` time (embedded in every generated `implementation.md`). If both stay active, the user will be asked about branch selection twice: once by `/sai-3-implement` (agent-level prompt) and once by `/sai-4-apply` (plan-level prompt). This is a documented trade-off — the plan-level prompt is self-contained and does not depend on agent memory of the earlier prompt. The sibling spec is NOT modified by this change.
- Option labels follow the user's input language (English fallback) — a user-provided constraint. The labels serve as UI-facing prompts within the plan.
