## Why

The current `audit-artifact-ingestion` behavior in `sai/instructions/implement.md` appends every actionable finding from `review.md` / `security.md` / `performance.md` / `accessibility.md` as code actions in `implementation.md`, with no editorial judgment. The result is implementation-plan bloat and code changes the user never asked for: imagine a 12-line review with 2 actionable findings and 10 minor nits — under the current behavior it produces 12 code-writing checkboxes. The fix is to make the agent exercise judgment over every finding using a fixed rubric, classify each as Apply (rendered as a code action) or Discard (listed in a Discarded sub-block with a one-sentence reason and surfaced in chat for confirmation), and apply this uniformly across all four audit artifacts — without breaking the re-run preservation contract that already exists.

## What Changes

- `sai/instructions/implement.md` — add a **Judgment rubric** section (five criteria: severity, actionability, spec-decision consistency, duplication, scope), add a **Discarded findings sub-block** to the audit step format inside `<plan_template>`, and exercise the judgment in both the first-run generation path and the re-run "Append audit-derived steps" path. The existing "Append audit-derived steps" wording is updated so every appended step contains the Apply code actions and the Discarded sub-block side by side, with the all-Discarded case producing a step with a single "No code changes from this audit" checkbox.
- `openspec/specs/audit-artifact-ingestion/spec.md` — replace the unconditional "extract all actionable findings and append" behavior with the Apply/Discard classification flow + new requirements for the rubric, Discarded block format, conversational confirmation, Q auto-discard with full transcription, and the all-Discarded closable-step case.
- `openspec/schemas/sai-workflow/schema.yaml` — one-line update to the `implementation` artifact description mentioning the Discarded findings block as part of the audit step format.

## Capabilities

### New Capabilities
- None. The judgment behavior is a new requirement set inside the existing `audit-artifact-ingestion` capability — the `audit-finding-judgment` sub-area is added via MODIFIED + ADDED Requirements on the same spec, not as a new capability. (Naming rationale: keeping it inside `audit-artifact-ingestion` keeps the per-artifact appending contract and the judgment contract in one spec — splitting them risks the two going out of sync.)

### Modified Capabilities
- `audit-artifact-ingestion`: the appending scenario is rewritten so the agent applies judgment and classifies every finding as Apply or Discard (the existing "extract all actionable findings and append" line is replaced); new requirements cover the five-criterion rubric, the Discarded-findings sub-block format, conversational confirmation in chat, automatic Q-finding discard with full text transcription, the all-Discarded closable-step case, uniform application across all four audit artifacts, and explicit preservation of the re-run preservation contract (no dedup).

## Additional Notes

- **The Discarded sub-block lives INSIDE the audit step**, not as a separate step. Each appended step (e.g., `Step N: Address review findings`) contains both the Apply code actions and the Discarded sub-block side by side. The sub-block format per entry: `**{id}** — {one-sentence reason} (source: {artifact} § {category} {id})`. The full Q text is transcribed verbatim beneath its entry.
- **Conversational confirmation** is in chat only — no `.openspec.yaml` approval key, no new gate. The user can override any discard by responding in chat before `/sai-4-apply` starts. The agent prints one line per discarded finding plus the literal Q text where applicable, then waits.
- **Q findings are auto-discarded with the reason `requires user response, not a code change`** because the existing audit-artifact template marks Qs as items needing clarification, not as code changes. The full Q text is transcribed so the user can answer in chat; the question is no longer "buried" in an audit artifact the next phase won't re-read. These Q findings are **post-implementation reviewer questions** (raised by `/sai-5-review`, `/sai-6-security`, `/sai-7-performance`, `/sai-8-accessibility`) and are **distinct from the design-phase Open Questions** governed by `openspec/specs/open-questions-gate/spec.md` — the two kinds of Q do not interact, but the new behavior only applies to the audit-artifact Qs.
- **Judgment is not idempotent w.r.t. the audit artifact**: re-running `/sai-3-implement` against the same unchanged `review.md` may produce a slightly different Apply/Discard list because the LLM is not perfectly deterministic. The re-run preservation contract still appends a new step per artifact; the contents of that step are best-effort, not byte-stable. This is a property of the LLM, not a defect in the rubric.
- **Re-run preservation contract is preserved**: re-running `/sai-3-implement` with the same audit artifact still appends a new step (numbered N+1) with its own Apply/Discard list. No dedup logic — the existing "Do NOT add audit-step dedup logic" comment in `implement.md` stays. The judgment is re-exercised on every re-run; if the user's chat-driven overrides changed the audit artifact, the new step reflects the new findings.
- **The all-Discarded case keeps the step closable**: the step still exists, contains only the Discarded sub-block and a single `- [ ] No code changes from this audit` checkbox, and the user closes it by checking the box. This avoids the "audit step vanished" surprise when every finding was discarded.
- **The five-criterion rubric is normative**: the spec invokes it by name and requires Apply/Discard classification to follow from the rubric outcome — not from gut feel. The five criteria are: (1) **severity** — does the finding rise to blocker/major or is it a non-issue? (2) **actionability** — is the proposed fix specific enough to implement? (3) **spec-decision consistency** — does the finding contradict a decision in `design.md` or a requirement in `specs/**/*.md`? (4) **duplication** — does the finding repeat another finding already addressed in an earlier step? (5) **scope** — does the finding stay within the change's scope or propose out-of-scope work?
- **The schema.yaml update is one line** — the existing `implementation` artifact description ("Granular implementation plan — code-level steps with RED→GREEN and STOP & COMMIT markers") gets a tail clause: "with Discarded findings sub-blocks inside audit steps derived from `review.md` / `security.md` / `performance.md` / `accessibility.md`". This keeps the schema description accurate so a future spec author reading the schema sees the Discarded sub-block as part of the contract.
- **No GLOSSARY.md needed**: no new domain terms are introduced. "Apply", "Discard", "Discarded findings sub-block", "judgment rubric", "conversational confirmation" are all new spec-defined terms living inside this spec; they don't enter the project glossary.
