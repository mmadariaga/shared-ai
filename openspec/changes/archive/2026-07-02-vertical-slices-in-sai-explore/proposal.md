## Why

The explore agent's prompt produced crystallization blocks that assumed every idea fit a single OpenSpec change. Large features got condensed into one oversized block, producing review-fatiguing diffs downstream. This change adds a slicing assessment before crystallization so the explore agent can recommend review-sized vertical slices (Walking Skeleton first, then dependency-ordered backlog) when the idea outgrows a single change, while keeping the existing single-block path for ideas that fit one change.

## What Changes

- `sai/instructions/explore.md` — adds a **Slicing assessment** step (item 2) that judges whether the idea fits a single change using three scope signals (orthogonal user tasks, no end-to-end path in ~1–3 days, more than one module boundary with no shared integration point).
- `sai/instructions/explore.md` — splits the crystallization protocol into two paths: the existing single-block protocol (renumbered 3) and a new **sliced** protocol (item 4) that emits one `Ready to Propose` block per slice, Walking Skeleton first, with `Depends on:` lines for backlog slices.
- `sai/instructions/explore.md` — renumbers the inline-proposal refusal to 5 and pluralizes the message ("block(s)") so it covers both the single-block and multi-block cases.

## Capabilities

### New Capabilities

- `explore-vertical-slicing`: explore agent assesses whether a candidate idea should crystallize as one OpenSpec change or as a Walking Skeleton plus dependency-ordered slices, and emits the matching block layout.

### Modified Capabilities

- `explore-context-isolation`: the inline-proposal refusal message becomes plural-aware ("block(s)") to cover the new multi-block sliced path; the crystallization protocol now has two variants (single change, sliced feature) but the no-inline invariant is preserved.

## Additional Notes

- The slicing decision is guidance, not a gate: if the user prefers to keep a large feature as one change, the agent states the review risk once, briefly, then uses the single-block path. The agent MUST NOT re-litigate the recommendation.
- The agent cannot see the eventual diff, so slicing signals are scope-based heuristics, not measurements. The "no end-to-end path in ~1–3 days" signal is the strongest of the three.
- Each emitted `Ready to Propose` block per slice is consumed by `/sai-1-spec` in its own new chat — the "new change per follow-up" pattern, not a single multi-change proposal.
- A "Walking Skeleton" in this prompt is the simplest end-to-end path that exercises every user task and is reversible; it is NOT a horizontal scaffold (e.g. types/interfaces only).
