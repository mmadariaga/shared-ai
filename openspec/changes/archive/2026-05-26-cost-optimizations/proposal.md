> **⚠ POST-HOC RECORD** — This proposal was backfilled after implementation. It describes a decision already made, not one being proposed.

## Why

The design and implement SAI instructions caused the main (frontier-tier) agent to perform token-expensive file I/O — reading source files, running glob/grep — work that a cheap haiku subagent can handle equally well. Similarly, the implement step embedded full test code inline in `implementation.md`, inflating the plan unnecessarily when a minimal stub plus a scenario list is sufficient for sai-4-apply to execute correctly.

## What Changes

- `sai/instructions/design.md` — codebase research fully delegated to explore subagent; main agent forbidden from file I/O on source code; Trust Rule added; tasks.md authoring rules tightened; Required Documentation gains a Spec files subsection; cost discipline reminder added.
- `sai/instructions/implement.md` — RED phase restricted to failing test + minimal stubs only; no logic that would make the test pass may appear in RED; tests in the plan expressed as stub + bullet scenario list for sai-4-apply to expand.

## Capabilities

### New Capabilities

- `design-subagent-delegation`: The main agent MUST delegate all codebase discovery and deep file reading to an explore subagent; it acts exclusively on that subagent's output.
- `tasks-authoring-rules`: tasks.md steps must describe HOW to build, reference spec files by path, and never restate requirements; Required Documentation must include a Spec files subsection populated from the explore subagent's report.
- `implement-red-phase-contract`: The RED phase in implementation.md may only contain the failing test, minimal stubs, and type-only scaffolding; full test assertions are deferred to sai-4-apply.

### Modified Capabilities

- None.
