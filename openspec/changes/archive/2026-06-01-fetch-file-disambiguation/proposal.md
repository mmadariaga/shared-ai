> **⚠ POST-HOC RECORD** — This proposal was backfilled after implementation. It describes a decision already made, not one being proposed.

## Why

Occasional issues loading instructions due to the LLM confusing `@sai/commands/X.md` with `@commands/X.md` — both patterns resolve to different directories but the agent had no explicit guidance to distinguish them.

## What Changes

- `skills/claude/fetch/SKILL.md`: Added a "File disambiguation" section clarifying that `@sai/commands/X.md` resolves to `~/.claude/sai/commands/X.md` while `@commands/X.md` resolves to `~/.claude/commands/X.md`.
- `skills/opencode/fetch/SKILL.md`: Added the same disambiguation section with opencode-equivalent paths (`~/.config/opencode/sai/commands/X.md` vs `~/.config/opencode/commands/X.md`).

## Capabilities

### New Capabilities

- `fetch-path-disambiguation`: Explicit disambiguation table preventing the agent from treating `@sai/commands/` and `@commands/` as equivalent namespaces.

### Modified Capabilities

- `claude-fetch-resolution`: Extended with a disambiguation subsection under the existing resolution rules.
- `wrapper-fetch-paths`: Clarified the distinct directory targets that wrappers reference.

## Impact

- **New files**: none
- **Modified files**: `skills/claude/fetch/SKILL.md`, `skills/opencode/fetch/SKILL.md`
- **Out of scope**: `design.md`, `tasks.md`, `implementation.md` — not generated by `/sai-backfill`
