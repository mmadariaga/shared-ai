# Proposal: design-emits-interfaces-artifact

## Why

REDâ†’GREEN proves a test is non-vacuous, but not that it encodes the spec rather than mirroring the implementation. Separating the contract (public signatures + the exact assertions to be tested) from its consumption is the precondition for a later change where a blind test author writes tests from the signature alone â€” closing the codeâ†’test mirroring channel. This slice (1 of 2) establishes and validates the artifact format in isolation, as a small reviewable unit, before anything wires it up.

## What Changes

- `sai-2-design` produces a new per-change artifact, `interfaces.md`, alongside `design.md` and `tasks.md`, written directly by the shared design instruction (`sai/instructions/design.md`).
- `interfaces.md` is keyed by `## Step N` using the same integer keying as `tasks.md` (and `implementation.md`). For each step it lists:
  1. the **new or modified public interfaces** (signatures) introduced in that step, and
  2. the **exact test assertions** for that step, each anchored to a `specs/**/*.md` requirement or scenario.
- The artifact is **consumable without seeing the implementation body** â€” signature + expected assertions are sufficient to author tests.
- `interfaces.md` is registered as an artifact in `openspec/schemas/sai-workflow/schema.yaml` with `requires: [tasks]`, **excluded from `apply.requires`** (no consumer in this slice), so `openspec` tooling recognizes it without gating apply on it.
- **No stage consumes it yet.** `sai-3-implement` and `sai-4-apply` are unchanged.

Non-goals:
- No consumer wiring â€” `sai-3-implement` respecting these interfaces and splitting `sai-4-apply` into a blind test-writer + implementation subagent is **slice 2** (`interface-first-test-isolation`).
- No testing-setup / stack section in `interfaces.md` â€” the testing stack stays single-sourced in `tasks.md`'s `## Implementation Context`; routing it to the test author is slice 2's job.
- No changes to `tasks.md`'s format â€” concrete signatures and exact assertions are the "detailed behavior" that `tasks.md`'s conciseness rule explicitly excludes, which is precisely why they live in a separate file.
- No change to the `sai-2-design` completion decision summary (it stays derived from `design.md` and `tasks.md`).

## Capabilities

### New Capabilities

- `design-interfaces-artifact`: `sai-2-design` emits `interfaces.md`, keyed by `## Step N` to `tasks.md`; per step it lists new/modified public signatures plus exact test assertions anchored to `specs/**/*.md`; it is a separate file from `tasks.md` that duplicates no testing-stack context and splits cleanly from `tasks.md`'s high-level `Testing Strategy`; it is regenerated wholesale alongside `tasks.md` each run (no cross-run preservation, so keys never desync); the artifact is registered in the schema (`generates: interfaces.md`, `requires: [tasks]`) but excluded from `apply.requires`; no stage consumes it in this slice.

### Modified Capabilities

(none â€” `interfaces.md` is an additive artifact. `design-instruction` still produces `design.md` and `tasks.md`; `tasks-scaffold-format` is untouched because the concrete signatures and assertions are deliberately kept OUT of `tasks.md`; `apply` is untouched because `interfaces.md` is excluded from `apply.requires`.)

## Impact

- **New files**: `openspec/changes/{name}/interfaces.md` (per change, generated by `sai-2-design`).
- **Modified files**:
  - `sai/instructions/design.md` â€” add an `interfaces.md` generation section after `tasks.md` generation.
  - `openspec/schemas/sai-workflow/schema.yaml` â€” register the `interfaces` artifact (`requires: [tasks]`, NOT in `apply.requires`); optional `templates/interfaces.md`.
- **No changes** to `sai-3-implement`, `sai-4-apply`, the audit phases, the completion decision summaries, or `apply.requires`.
- **Interface-before-consumer risk (accepted)**: the format's real sufficiency is only validated once slice 2 wires a consumer. This slice validates the format in isolation via review.
- **Interface-before-implementation churn (accepted, flagged for slice 2)**: requiring concrete signatures at design time â€” before any code exists â€” pulls real interface design forward. Some signatures authored here will prove wrong once implementation starts, and that churn lands in slice 2 (where the blind test author depends on them). Slice 2's spec should inherit this concern explicitly.

## Additional Notes

- Step N keying is shared across `tasks.md`, `implementation.md`, and now `interfaces.md`; all three are generated/derived from the same step decomposition, so keys align at generation time.
- `sai-2-design` regenerates `design.md` and `tasks.md` from scratch each run and has no re-run preservation path (unlike `sai-3-implement`, which preserves `implementation.md` byte-for-byte). `interfaces.md` follows the design-stage model: regenerated wholesale each run from the same fresh decomposition as `tasks.md`, so orphaned or desynced `## Step N` sections cannot arise. The orphan-preservation mechanism in `implement.md` is deliberately NOT imported here â€” it would desync `interfaces.md` (carrying old steps) from a freshly regenerated `tasks.md`.
- The user confirmed schema registration in this slice (with `interfaces` excluded from `apply.requires`) so the artifact graph stays complete and `openspec status` recognizes the file before slice 2 adds a consumer.
