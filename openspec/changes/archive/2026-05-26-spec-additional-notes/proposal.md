## Why

The `proposal.md` format enforced a rigid structure where every piece of information had to fit into a named section (Why, What Changes, Capabilities, Impact, Research Documentation). Useful context gathered during spec research — existing object shapes, known gotchas, code snippets, implementation hints — had no home and was being silently dropped. Adding a free-form `## Additional Notes` section gives spec authors a place to capture non-normative information that doesn't belong in requirements or research links but is genuinely valuable for the designer and implementer.

## What Changes

- `openspec/schemas/sai-workflow/schema.yaml`: added `**Additional Notes**` bullet to the `artifacts.proposal` section description, listing it as a valid top-level section of the proposal artifact.
- `openspec/schemas/sai-workflow/templates/proposal.md`: added `## Additional Notes` section with a comment placeholder describing its intended use (non-normative facts: existing code, object shapes, gotchas not expressible as requirements).
