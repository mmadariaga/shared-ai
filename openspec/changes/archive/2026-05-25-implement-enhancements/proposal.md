> **⚠ POST-HOC RECORD** — This proposal was backfilled after implementation. It describes a decision already made, not one being proposed.

## Why

Running `/sai-3-implement` repeatedly (e.g. after partial progress) caused unnecessary cost and context bloat: the re-run guard (Step 1b) read and compared the full existing `implementation.md` inside the main agent context, and the research phase re-loaded all documentation even when only a subset of steps were new. Two targeted subagent hints were added to offload the expensive I/O steps.

## What Changes

- `sai/instructions/implement.md` — Step 1b receives a `> **Subagent:** Run this step in a separate subagent.` directive, moving the file-read and checkbox-scan work out of the main agent context.
- `sai/instructions/implement.md` — The research phase (Step 2) gains an exception: when re-running with a non-empty applied-steps set, a subagent scoped to only the new elements may be spawned for documentation research, overriding the blanket "no subagents for research" rule.

## Capabilities

### New Capabilities

- `rerun-guard-subagent`: Step 1b of the implement instruction runs in a dedicated subagent, isolating the existing-file read and checkbox-scan from the main agent context.
- `rerun-research-exception`: When re-running an already-partially-applied implementation plan, the research phase may spawn a subagent scoped to elements introduced since the last run, bypassing the normal no-subagent-for-research constraint.

### Modified Capabilities

None.
