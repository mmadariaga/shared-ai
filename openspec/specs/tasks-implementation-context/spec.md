## Requirements
### Requirement: tasks-implementation-context-section
`tasks.md` SHALL contain a `## Implementation Context` section generated by `/sai-2-design` immediately after `## Required Documentation`.

The section SHALL contain exactly three fields:

    **Stack**: <primary language/framework + key versions relevant to this change>

    **Conventions**: <2–5 project-specific, non-obvious bullets derived from codebase research>

    **Avoid**: <anti-patterns the implementation agent might default to, given the declared stack>

Each field SHALL be derived from codebase research performed during design, not copied from generic content or inferred from the change description alone.

The section SHALL NOT be omitted or left with placeholder text.

#### Scenario: sai-2-design derives context from codebase research

- **WHEN** `/sai-2-design` generates `tasks.md`
- **THEN** `## Implementation Context` contains Stack with concrete versions, Conventions with 2–5 bullets that reflect patterns observed in the actual codebase (not generic best-practices), and Avoid with anti-patterns specific to this stack that a model might default to

#### Scenario: sai-3-implement uses context as authoritative contract

- **WHEN** `/sai-3-implement` reads `tasks.md` to produce `implementation.md`
- **THEN** it adopts **Stack**, **Conventions**, and **Avoid** as a non-negotiable implementation contract and does not perform codebase exploration to recover stack or convention information

#### Scenario: conventions are project-specific, not generic

- **WHEN** `/sai-2-design` writes the Conventions field
- **THEN** each bullet describes a pattern specific to this project (e.g., a naming convention, file organization rule, or error-handling idiom observed in the codebase), not a universal best-practice such as "write readable code" or "follow SOLID principles"

### Requirement: tasks-implementation-context-replaces-expertise-profile
`## Implementation Context` SHALL replace the verbose Expertise Profile template (Primary Role, Technologies, Standards, Output Quality Bar subsections) that `/sai-3-implement` previously expected from the loaded `plan.md`.

`/sai-3-implement` SHALL read **Stack**, **Conventions**, and **Avoid** from `tasks.md` as the equivalent of the Expertise Profile. It SHALL NOT require the user to supply a separate Expertise Profile, and SHALL NOT STOP when the Expertise Profile subsections are absent from other artifacts.

Generic "Output Quality Bar" and universal standards SHALL remain in `plan.md` as hard rules and SHALL NOT be duplicated into `tasks.md`.

#### Scenario: sai-3-implement proceeds without separate Expertise Profile

- **WHEN** `/sai-3-implement` reads a `tasks.md` containing `## Implementation Context` with Stack, Conventions, and Avoid
- **THEN** it treats those three fields as the complete Expertise Profile contract and does not STOP to request additional profile information

#### Scenario: sai-3-implement stops when implementation context is missing

- **WHEN** `/sai-3-implement` reads a `tasks.md` that lacks `## Implementation Context` entirely
- **THEN** it STOPs and asks the user to run `/sai-2-design` again or supply the missing context manually

### Requirement: tasks-implementation-context-schema
The `tasks` artifact instruction in `openspec/schemas/sai-workflow/schema.yaml` SHALL include a directive requiring `/sai-2-design` to populate `## Implementation Context` from codebase research, with explicit guidance that generic content is not acceptable.

#### Scenario: schema instruction prohibits generic implementation context

- **WHEN** `openspec instructions tasks --change <name>` is run
- **THEN** the returned `instruction` field states that Stack, Conventions, and Avoid MUST be derived from the actual codebase and that generic or placeholder content is not acceptable
